## Design Context

### Users

**Audience: Keith, sole user.** This is a personal deploy admin — not a multi-tenant SaaS. There is no onboarding to design for, no permissions matrix to negotiate, no "first-run hand-holding" tax. The interface is used by someone who built the system, knows what every button does, and wants to get to work fast.

**Context of use:** Checked at odd hours from a desk, a couch, or a phone. Often during the small dopamine moment of "I just pushed code, did the site deploy?" Sometimes in a frustrated "why is the site down?" mode at 11pm. The interface should reward both moods.

**Job to be done:** Paste a Git URL → get a live site. Beyond that — monitor, redeploy, tail logs, tweak env vars, kill misbehaving containers. Operationally simple. Emotionally, it should feel like *coming home* to a personal tool, not logging in to a cloud control panel.

### Brand Personality

**Three words: warm, playful, opinionated.**

Built in the wake of Glitch's shutdown. The product carries a bit of grief and a lot of inheritance — Glitch's spirit of remix-culture, friendly copy, bright color, and the sense that a deploy platform could be a place rather than a service. This is *not* a Glitch tribute or a cosplay; it's a personal tool that took the parts of Glitch that mattered and rebuilt them in TypeScript and Bun.

**Tone of voice:** First-person and casual where it makes sense ("your sites", "let's go"). Empty states, error messages, and edge-case copy should feel like a friend talking, not a help-desk script. A little irreverent. Never corporate. Never "We're sorry, something went wrong."

**Emotions to evoke:** Cozy competence. The satisfaction of opening a personal terminal. A flicker of nostalgia for the old web. The confidence of "this is mine, and it works."

### Aesthetic Direction

**Direction: terminal-meets-toy.** Utilitarian structural bones (hard borders, monospace typography, dense data) with playful, personality-rich details (color, motion, copy, the occasional unexpected flourish). The structural rigor is what makes the playful moments land — without it, "playful" becomes "noisy."

**Theme:** Both light and dark, derived from `prefers-color-scheme` with manual toggle. Both themes must feel *intentional and equally cared-for*, not "dark mode is the canonical one and light is an inverted afterthought." Light theme is for daytime work; dark theme is the late-night admin session.

**Color:** The hot pink (`#e91e8c`) is currently the lone accent — open to evolving. Recommended direction is a small, expressive Glitch-inheritance palette: a primary hot pink/magenta, paired with **one or two characterful supports** (a soft cyan, a warm yellow, a moss green — to be decided in implementation). 60-30-10 still applies; the supporting colors should be rare, used to differentiate state (success / building / sleeping / error) with personality, not just generic green/yellow/red. Tint neutrals subtly toward the brand magenta so the surfaces feel cohesive instead of "white plus pink."

**Typography:** Monospace-first, terminal-rooted — but **not JetBrains Mono.** JetBrains Mono is the trained-reflex monospace and gives the admin a generic "developer tool" voice instead of a personal one. Direction is a more distinctive monospace with character: candidates like Departure Mono, Monaspace Argon/Radon, Cartograph CF, GT Maru Mono, or PP Neue Bit. The bundled `PPNeueMontrealMono-*.woff2` files in `packages/admin/fonts/` appear to be unused legacy assets and should likely be removed or replaced with the chosen face. Consider pairing the body monospace with a *second* display face (sans, slab, or something weirder) used sparingly for page titles and empty-state hero text — this is where personality lives.

**Layout:** Keep the existing dense, list-based, hard-bordered structure. It's the right bones for an operator's tool. Vary spacing and break the grid intentionally for emphasis (page headers, empty states, the big moments) — don't apply the same padding everywhere. Avoid card-grid templating; the current site list is correctly a list, not a grid of identical tiles.

**Anti-references — explicitly NOT:**
- Vercel / Linear / Railway / Render / Netlify dashboards
- Generic SaaS admin (sidebar + top bar + card grid + chart widget)
- Dark mode with cyan/purple glowing accents over near-black
- Glassmorphism, blur effects, gradient hero text
- Border-left colored accent stripes on cards or list items
- Sparklines and metric tiles used as decoration
- Anything that reads as "AI-generated dashboard, 2024"

### Design Principles

1. **Terminal bones, toy soul.** Hard borders, monospace, dense data. Playfulness comes from color, copy, motion, and the occasional unexpected detail — never from softening the structure with rounded corners or pastel cards.
2. **A personal place, not a product.** Copy is first-person and casual. Empty states and errors talk like Keith would, not like a help-desk template. No "Welcome!" "Get started!" "We" pronouns.
3. **Glitch as inheritance, not pastiche.** Borrow the *spirit* (warmth, remix energy, expressive color, friendly tone) — never the literal aesthetic. If a choice feels like cosplay, reject it.
4. **Both themes are first-class.** Every component must be designed in light *and* dark, both feeling deliberate. No "dark is real, light is an export."
5. **Color is the personality, not the decoration.** The palette (magenta + small set of supports) is the brand. Use it sparingly, use it consistently, use it to convey state with character.
6. **Distinctive type, not "developer mono."** The monospace must have a voice. Pair with a display face for the moments that matter. JetBrains Mono is banned from this project.
7. **Density is a feature.** This is an operator's tool. Don't pad it into a marketing site. Lists over card grids. Information over chrome.
8. **The bans are real.** No border-left accent stripes. No gradient text. No pure black/white. No glassmorphism. No identical-card grids. No "AI dashboard 2024" tells.

## Design Tokens

Concrete values for the principles above. These are the canonical choices — future craft sessions should apply them rather than re-deciding. All values are starting points; tune in implementation, but don't drift from the conceptual direction.

### Typography

**Body / data: Monaspace Argon** (GitHub Next, SIL OFL, self-hosted via `@font-face`)
- Weights: 400 (regular), 500 (medium), 700 (bold)
- Used for: all UI text, log lines, env vars, button labels, navigation, form fields
- Chosen for humanist-mono warmth at small sizes without losing operator-tool credibility

**Display: Departure Mono** (Helena Zhang, OFL, self-hosted via `@font-face`)
- Weights: 400 only
- Used for: page titles, empty-state hero text, big deploy moments, occasional flourishes
- Used at 24px+ only — Departure's CRT character degrades at body sizes
- This is where the Glitch-era nostalgia lives

**Banned in this project:** JetBrains Mono, IBM Plex Mono, Space Mono, Roboto Mono, Fira Code, system mono fallbacks as primary face. Generic system-ui sans is also banned for body — we commit to monospace.

### Color (OKLCH, mode-agnostic)

```css
/* Brand palette */
--brand-magenta:  oklch(63% 0.26 0);      /* signature, refines #e91e8c */
--brand-cyan:     oklch(76% 0.14 210);    /* Glitch-era support */
--brand-yellow:   oklch(86% 0.16 88);     /* Glitch-era support */

/* State (use brand colors deliberately, not generic R/Y/G) */
--state-running:  oklch(72% 0.13 145);    /* moss-sage */
--state-building: oklch(86% 0.16 88);     /* = brand-yellow */
--state-sleeping: oklch(72% 0.05 305);    /* lavender-grey */
--state-error:    oklch(60% 0.18 25);     /* warm clay red */
```

Light theme neutrals (tinted toward magenta hue 350°):

```css
--bg:       oklch(99% 0.003 350);
--surface:  oklch(97% 0.006 350);
--hover:    oklch(95% 0.008 350);
--border:   oklch(92% 0.008 350);
--border-strong: oklch(82% 0.012 350);
--text:     oklch(20% 0.01  350);
--muted:    oklch(50% 0.012 350);
--faint:    oklch(70% 0.008 350);
```

Dark theme neutrals (same hue, inverted lightness):

```css
--bg:       oklch(13% 0.005 350);
--surface:  oklch(17% 0.008 350);
--hover:    oklch(22% 0.01  350);
--border:   oklch(28% 0.012 350);
--border-strong: oklch(40% 0.015 350);
--text:     oklch(95% 0.005 350);
--muted:    oklch(65% 0.012 350);
--faint:    oklch(45% 0.012 350);
```

**Cohesion rule:** Neutrals are tinted ~10° cooler than the magenta (which sits at 0°). The whisper of warm-magenta in greys is what makes brand-on-surface feel native instead of pasted-on. Don't change the hue across themes.

**Accent usage rule (60-30-10 by weight, not pixel count):**
- 60% surface/text neutrals
- 30% secondary text and borders
- 10% brand-magenta — this includes hover states, focus rings, the active nav item, primary buttons, and `<a>` hover. The pink is rare-by-design; that's what makes it work.
- Cyan and yellow appear *only* via state (running, building) — they are not free-floating accents. Sleeping uses lavender-grey. Don't reach for cyan/yellow for "emphasis."

### Spacing

Keep the existing 4pt scale (`--space-1` through `--space-8` = 4/8/12/16/24/32/48/64). Add `--space-9: 96px` for empty-state hero moments. Rename to semantic tokens (`--space-xs/sm/md/lg/xl/2xl/3xl/4xl/5xl`) only if a craft session is already touching those declarations — don't churn for the rename alone.

### Type scale

Keep current fixed-`px` scale (this is app UI, not marketing). Increase contrast between display and body:

```css
--text-xs:    11px;  /* timestamps, faintest labels */
--text-sm:    12px;  /* secondary body */
--text-base:  14px;  /* primary body */
--text-lg:    16px;  /* emphasized body */
--text-xl:    20px;  /* small headings */
--text-2xl:   28px;  /* page titles — bumped from 24, switches face to Departure Mono */
--text-3xl:   40px;  /* empty-state hero — Departure Mono */
--text-display: clamp(48px, 8vw, 96px);  /* "big moment" — Departure Mono */
```

The face-switch at `--text-2xl` is the key move: small text is workhorse Argon, large text becomes character-rich Departure. That's the "terminal bones, toy soul" principle made literal.

### Borders & corners

Hard 1px borders, **`border-radius: 0`** everywhere. Rounded corners undermine the terminal-bones principle. The single concession: status dots (the 10px circular site-status indicators) stay round — they're a different visual category.

### Motion

Default duration: 150ms (matches existing). Reserve longer durations (300–500ms) for the deploy-success moment, page transitions, and modal entry. Use exponential easing (`cubic-bezier(0.16, 1, 0.3, 1)` — ease-out-quint) for anything > 200ms. No bouncy/elastic. Respect `prefers-reduced-motion`.

### Voice & UX writing

- First-person and casual: "your sites", "let's deploy this", "nothing here yet — paste a Git URL above"
- Errors describe what happened from your perspective, not "We're sorry": "couldn't reach the git host" / "build failed — check the logs"
- Empty states teach the next action; never just "no items"
- Button labels are verbs, not nouns: "deploy", "redeploy", "tail logs", "kill", not "Deployment" / "Redeployment"
- Lowercase by default for buttons and labels (existing uppercase utility-label pattern stays for table headers and section titles — that's a different category and already working)
